# 硬件加法乘法消耗

## 在硬件电路里，做乘法是非常昂贵且耗电的. 为什么gpu矩阵乘法很快？

这是一个非常敏锐且深刻的观察！你触及了计算机体系结构中**“计算密度”**与**“算力效率”**之间最核心的矛盾。

之所以觉得“矛盾”，是因为你忽略了一个关键变量：**任务的性质（Scalar vs. Tensor）**以及**硬件的设计目标（Latency vs. Throughput）**。

我们可以从以下三个维度来拆解这个看似矛盾的现象：

### 1. “昂贵”的定义不同：延迟（Latency）vs 吞吐量（Throughput）

当我们说“乘法很昂贵”时，主要指的是在**标量运算（Scalar Operation）**或**单指令流**中的**时间成本（Latency）**和**逻辑门面积（Area）**。

*   **在 CPU/CORDIC 场景下（追求低延迟）：**
    这里的目标是“尽快算出一个结果”。如果你要在一条流水线上完成 $\sin(x)$ 的计算，每一个乘法器都要占用宝贵的逻辑门空间。如果每个时钟周期都要进行一次复杂的 $n$ 位浮点乘法，电路会变得极其庞大、发热严重，且由于信号传输路径长（Critical Path），会导致主频无法提升。
    **CORDIC 的哲学是：用“廉价”的移位和加法，通过多次迭代，换取一个“昂贵”的乘法结果。**

*   **在 GPU/矩阵乘法场景下（追求高吞吐量）：**
    GPU 的目标不是“单次计算多快”，而是“单位时间内算多少个”。GPU 并不关心某一个乘法用了多少纳秒，它关心的是在同一个时钟周期内，能不能塞进一万个乘法器。

### 2. 硬件结构的差异：逻辑电路 vs 专用阵列 (Systolic Array)

这是最根本的技术差异。

*   **通用计算（CPU/CORDIC）：**
    它是**“逻辑驱动”**的。每一个乘法操作都需要经过：取指令 $\rightarrow$ 译码 $\rightarrow$ 从寄存器读数 $\rightarrow$ 进入 ALU $\rightarrow$ 写回结果。这个过程中的数据搬运和控制逻辑开销，远大于乘法本身。在处理三角函数这种逐个计算的任务时，这种“管理成本”显得非常昂贵。

*   **矩阵计算（GPU/TPU）：**
    它是**“结构驱动”**的。现代 GPU 核心（如 NVIDIA 的 Tensor Core）使用的是**脉动阵列（Systolic Array）**或高度优化的乘加单元（FMA）。
    在矩阵乘法中，数据不是“算完一个丢掉一个”，而是像血液流经血管一样，在计算单元阵列中**流动**。一个数被读入后，会与一整行/一整列的数据连续进行乘加运算。
// **这种设计极大地降低了“访存开销（Memory Access Overhead）”。** 矩阵乘法之所以快，不是因为单个乘法器变便宜了，而是因为我们通过高度重复利用数据，抵消了由于大规模并行带来的巨大搬运成本。

### 3. 算力密度的权衡：面积与能耗

我们可以用一个物理学类比：

*   **CORDIC 就像是“手工雕刻”：** 你不需要复杂的工具（只需要简单的凿子），但你只能在一个点上慢慢刻，速度慢，适合处理精细、单个的形状（标量函数）。
*   **GPU 矩阵运算就像是“模具注塑”：** 你需要建立一套极其昂贵、庞大的模具系统（Tensor Core 的大量晶体管），虽然初期建设成本极高（面积大、耗电猛），但一旦开机，你可以一秒钟生产出成千上万个完全相同的零件（矩阵元素）。

### 总结

| 维度 | CORDIC (处理三角函数) | GPU Tensor Core (处理矩阵) |
| :--- | :--- | :--- |
| **核心矛盾** | 如何在不使用乘法器的情况下算准 | 如何在极高带宽下喂饱成千上万个乘法器 |
| **关注点** | **减少面积与能耗**（避开昂贵的乘法） | **掩盖延迟**（通过并行化抵消访存成本） |
| **数据特征** | 逐个处理，逻辑依赖性强 | 大规模重复，数据复用率极高 |
| **结论** | 在小规模、低功耗场景下，乘法是**负担** | 在大规模、高性能场景下，乘法是**引擎** |

所以，**“乘法很昂贵”这句话在逻辑电路设计层面永远成立；而“矩阵乘法很快”在并行计算架构层面也永远成立。两者并不冲突，只是它们在解决不同的“贫穷”问题。**

## 从逻辑电路来说， 加法为什么比乘法便宜

这是一个非常经典的数字电路设计问题。在逻辑门（Logic Gates）的层面上，“便宜”主要体现在两个维度：**晶体管数量（面积/Cost）**和**信号传播延迟（速度/Delay）**。

我们可以从以下三个层面来理解为什么加法比乘法“廉价”得多：

### 1. 逻辑复杂度的本质差异（计算层级）

从数学运算的结构来看，乘法本质上是**“一连串的移位和加法”**。

*   **加法（Addition）：** 它的操作对象是两个数字的对应位。对于 $n$ 位加法，你只需要考虑每一位与其进位（Carry）的关系。逻辑关系是**线性的**，每个比特位的计算只依赖于前一位的状态。
*   **乘法（Multiplication）：** 如果你要计算两个 $n$ 位数相乘，其结果最高可达 $2n$ 位。从底层来看，乘法器需要完成两件事：
    1.  **部分积生成（Partial Product Generation）：** 你需要将第一个数的每一位，分别与第二个数的每一位进行“与（AND）”运算。这会产生 $n^2$ 个逻辑项。
    算术层面上，乘法是一个**二维**的操作过程，而加法是一个**一维**的操作过程。

### 2. 晶体管数量与面积（Cost/Area）

在数字电路中，面积就是金钱。

*   **加法器（Adder）：** 一个全加器（Full Adder）只需要几个基本的逻辑门（异或 XOR、与 AND、或 OR）。实现一个 $n$ 位加法器，其晶体电容和逻辑门的数量随 $n$ 呈**线性增长 ($O(n)$)**。
*   **乘法器（Multiplier）：** 即使是现代最先进的 **Wallace Tree（华莱士树）** 或 **Dadda Multiplier** 结构，虽然它通过复杂的树状结构优化了计算过程，但它本质上仍然需要处理那 $n^2$ 个部分积。
    随着位数 $n$ 的增加，乘法器所需的逻辑门数量呈**平方级增长 ($O(n^2)$)**（虽然经过树状压缩可以降到接近 $O(n \log n)$，但依然远高于加法的线性增长）。

**结论：** 同样的位宽下，乘法器占用的芯片面积通常是加法器的数倍甚至数十倍。

### 3. 信号传播延迟与进位链（Delay/Latency）

在电路中，电流通过逻辑门需要时间。

*   **加法的挑战（进位传播）：** 加法最难的地方在于“进位传递”（Carry Propagation）。低位的进位必须传到高位，这会形成一条长长的路径。虽然我们有“超前进位加法器（CLA）”来加速，但随着位数增加，信号传递的延迟依然存在。
*   **乘法的挑战（多级压缩）：** 乘法不仅面临加法那样的进位问题，还面临着**“部分积压缩”**的问题。你需要把成百上千个逻辑项，通过层层树状结构（Reduction Tree）压缩到最终的两个操作数。每一层压缩都需要经过多级门电路，信号经过的路径深度（Logic Depth）远比加法器深得多。

### 总结对比图

| 特性 | 加法器 (Adder) | 乘法器 (Multiplier) |
| :--- | :--- | :--- |
| **数学本质** | 一维线性操作 | 二维矩阵/部分积操作 |
| **逻辑规模** | $O(n)$ —— 随位数线性增长 | $\approx O(n^2)$ 或 $O(n \log n)$ —— 爆炸式增长 |
| **电路结构** | 链式或树状进位 | 大规模部分积生成 + 复杂的压缩树 |
| **物理成本** | 极低（晶体管少，面积小） | 高（需要大量逻辑门来处理交叉项） |

**这就是为什么在硬件设计中，工程师会绞尽脑汁地把乘法转化成加法（如 CORDIC 算法、Booth 编码），因为“逃避”乘法就是直接节省成本和降低功耗。**
